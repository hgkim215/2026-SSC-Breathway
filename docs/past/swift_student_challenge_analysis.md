## 🎯 행사 개요

### Swift Student Challenge란?

Swift Student Challenge는 Apple이 전 세계 학생 개발자들을 대상으로 매년 개최하는 **무료 코딩 챌린지**입니다. 학생들이 자신의 창의성과 프로그래밍 실력을 Swift와 Xcode를 활용하여 선보일 수 있는 기회를 제공합니다.

### 핵심 특징

- **무료 참가**: 별도의 참가비 없음
- **글로벌 대회**: 전 세계 학생 대상
- **실용적 경험**: 실제 앱 개발 프로세스 체험
- **Apple과의 직접 연결**: 우수 수상자는 Apple Park 방문 기회

### 대회의 목적

1. **차세대 개발자 육성**: 미래의 개발자, 창작자, 기업가 양성
2. **기술력 향상**: Swift 및 Apple 플랫폼 개발 능력 강화
3. **창의성 발휘**: 독창적인 아이디어를 코드로 구현
4. **사회적 가치 창출**: 지역사회와 세계의 문제 해결

## 📦 제출 요구사항

### 기술적 요구사항

#### 1. 제출 형식

```
파일 형식: .swiftpm (App Playground)
압축 형식: ZIP 파일
최대 용량: 25MB
```

#### 2. 개발 환경

- **Xcode 16** 이상 또는
- **Swift Playgrounds 4.6** 이상

두 가지 방법으로 프로젝트 생성 가능:

**Xcode에서:**

```
Xcode > Create New Project > iOS > App Playground
```

**Swift Playgrounds에서:**

```
Swift Playgrounds > Create New App
```

#### 3. 실행 시간

- **3분 이내**에 앱의 핵심 기능과 가치를 경험할 수 있어야 함
- 심사위원이 해당 시간 내에 앱을 완전히 평가할 수 있어야 함

#### 4. 네트워크 및 리소스

- ❌ **네트워크 연결 의존 금지** (오프라인에서 심사)

- ✅ Apple Pencil 활용 가능

#### 5. 언어

- 모든 콘텐츠는 **영어**로 작성

## 🏆 평가 기준

Apple은 다음 4가지 핵심 가치를 기준으로 평가합니다:

### 1. 혁신성 (Innovation)

- 독창적인 아이디어와 접근 방식
- 기존에 없던 새로운 솔루션 제시
- 기술의 창의적 활용

**평가 포인트:**

- 기존 앱과 차별화되는 요소가 있는가?
- 문제 해결 방식이 독창적인가?
- Apple의 최신 기술(SwiftUI, SwiftData, Vision 등)을 창의적으로 활용했는가?

### 2. 창의성 (Creativity)

- 예술적 표현과 디자인
- 사용자 경험의 독창성
- 스토리텔링 능력

**평가 포인트:**

- 시각적 디자인이 매력적이고 일관성 있는가?
- 사용자 인터랙션이 창의적이고 직관적인가?
- 앱이 감정적으로 사용자와 연결되는가?

### 3. 사회적 영향 (Social Impact)

- 지역사회나 글로벌 이슈 해결
- 실질적인 문제에 대한 솔루션 제공
- 긍정적인 변화 창출

**평가 포인트:**

- 실제 사람들에게 도움이 되는가?
- 사회적으로 의미 있는 문제를 다루는가?
- 접근성과 포용성을 고려했는가?

### 4. 포용성 (Inclusivity)

- 다양한 사용자 고려
- 접근성 기능 구현
- 문화적 감수성

**평가 포인트:**

- VoiceOver, Dynamic Type 등 접근성 기능 지원
- 다양한 배경의 사용자를 고려한 디자인
- 언어, 문화, 능력에 관계없이 사용 가능

## 💡 수상 가능성을 높이는 전략

### 1. 강력한 아이디어 선정

#### ✅ 좋은 아이디어의 특징

- **개인적 경험에서 출발**: 본인이 직접 겪은 문제나 관찰한 이슈
- **명확한 타겟**: 누구를 위한 앱인지 명확
- **측정 가능한 임팩트**: 어떤 변화를 만들 수 있는지 구체적
- **실현 가능성**: 3분 안에 핵심 가치 전달 가능

#### 📌 아이디어 브레인스토밍 질문

1. 내 주변에서 사람들이 겪는 불편함은 무엇인가?
2. 내가 열정을 가진 분야에서 개선할 점은?
3. 내 가족, 친구, 지역사회가 필요로 하는 것은?
4. 접근성이나 포용성 측면에서 개선 가능한 영역은?

### 2. 사용자 중심 디자인

#### Human Interface Guidelines 준수

- [Apple HIG](https://developer.apple.com/design/human-interface-guidelines) 철저히 학습
- 플랫폼별 디자인 패턴 이해
- 일관성 있는 UI/UX 제공

#### 접근성 우선 설계

```swift
// 예: VoiceOver 지원
Text("Welcome")
    .accessibilityLabel("Welcome to the app")

// Dynamic Type 지원
Text("Title")
    .font(.title)
    .dynamicTypeSize(.large)
```

### 3. 기술적 탁월성

#### Apple 최신 기술 활용

**2026년 추천 기술 스택:**

- **SwiftUI**: 현대적이고 선언적인 UI 구축
- **SwiftData**: 데이터 관리
- **Swift Charts**: 데이터 시각화
- **RealityKit**: AR/VR 경험
- **Vision Framework**: 컴퓨터 비전
- **Core ML**: 머신러닝 통합
- **WidgetKit**: 위젯 제공
- **App Intents**: Siri 통합

> [!TIP]
> WWDC 2025의 "What's new in Xcode 16" 세션을 반드시 시청하세요. 최신 기능과 best practices를 학습할 수 있습니다.

#### 코드 품질

- 깔끔하고 읽기 쉬운 코드
- 적절한 주석 (특히 복잡한 로직)
- Swift 코딩 컨벤션 준수
- 에러 처리 구현

### 4. 스토리텔링

#### 앱의 서사 구조

1. **문제 제시**: "왜 이 앱이 필요한가?"
2. **솔루션 소개**: "어떻게 문제를 해결하는가?"
3. **가치 전달**: "사용자에게 어떤 변화를 주는가?"

**앱 개요:**

- 응급 상황(출혈, 화상, 질식 등)에 대한 단계별 응급처치 가이드 제공
- Panic Mode 기능: CPR 등을 음성으로 안내하여 hands-free 작동

**성공 요인 분석:**

#### 1. 강력한 개인적 동기

- 캠퍼스에서 발작하는 사람을 목격한 경험
- "아무도 무엇을 해야 할지 몰랐다"는 실제 문제 인식

- 응급 상황에서 즉시 접근 가능한 정보 제공

#### 3. 사용자 경험 혁신

- Panic Mode: 음성 내레이션으로 hands-free 작동
- 응급 상황의 특수성 고려 (손을 쓸 수 없는 상황)

#### 4. 빠른 개발 실행력

- 15일 만에 완성 (집중력과 효율성)

**배울 점:**

- ✅ 실제 경험에서 나온 진정성 있는 문제 해결
- ✅ 명확한 사회적 가치 제시
- ✅ 사용 상황에 맞는 UX 설계

### 사례 2: Yume's Spellbook by Larissa Okabayashi (브라질)

**앱 개요:**

- 비전문가를 위한 대규모 언어 모델(LLM) 교육 앱
- 애니메이션 스타일의 친근한 디자인

**성공 요인 분석:**

## 🚀 최종 조언

### DO ✅

1. **진정성 있는 문제 선택**
   - 본인이 진심으로 관심 있는 주제
   - 실제 사람들에게 도움이 되는 솔루션

2. **단순하지만 완성도 높게**
   - 많은 기능보다 핵심 기능의 완벽한 구현
   - 짧은 시간 안에 "와우" 하게 만들기

3. **사용자 테스트**
   - 최소 5명 이상에게 피드백 받기
   - 실제 사용 시나리오로 테스트

4. **디테일에 신경쓰기**
   - 애니메이션, 피드백, 에러 메시지까지 섬세하게
   - 접근성 기능 필수 구현

5. **스토리 전달**
   - 왜 이 앱을 만들었는지 명확히 설명
   - 사용자의 삶에 어떤 변화를 주는지 보여주기

### DON'T ❌

1. **마지막 순간까지 미루지 말기**
   - 마감일 당일 제출은 위험
   - 예상치 못한 문제 대비

2. **복잡하게 만들지 말기**
   - 너무 많은 기능은 오히려 독
   - 핵심에 집중

3. **네트워크 의존하지 말기**
   - 오프라인에서 완벽히 작동해야 함
   - 모든 리소스 로컬에 포함

4. **표절하지 말기**
   - 오픈소스 사용 시 반드시 크레딧 명시
   - 개인 작업물임을 증명

5. **포기하지 말기**
   - 수상 여부와 상관없이 귀중한 경험
   - 3번까지 도전 가능

## 🫁 호흡기 환자 예방 기반 작품 아이디어 제안 (Problem Definition 적용)

### 참고 영상

- https://www.youtube.com/watch?v=wk6WIlWPdU0

### Problem Brief: Preventable Breathing Crises in Daily Life

## 1. Decision Context (Owner / Deadline / Why Now)

- Decision: Swift Student Challenge 2026 제출 주제를 `호흡기 환자의 예방 행동 + 위기 시 소통` 문제로 확정할지 결정
- Owner: Hyeongik Kim
- Decision Deadline: 2026년 2월 20일
- Why Now: SSC 제출 마감일(2026년 2월 28일) 전 최소 1주일의 구현/테스트 시간이 필요함

## 2. Problem Summary (Current vs Desired Gap)

- Current: 호흡기 환자와 보호자는 예방 수칙을 "알고 있어도" 일상에서 꾸준히 실천하기 어렵고, 위기 순간에는 소통 비용이 급격히 증가함
- Desired: 일상 예방 행동이 작고 반복 가능한 루틴으로 정착되고, 위기 순간에도 최소한의 입력으로 도움 요청 및 대응 정보 공유가 가능해야 함

## 3. Target Users and Stakeholders

- Primary User: COPD 등 만성 호흡기 질환 환자
- Secondary User: 가족 보호자
- Stakeholders: 재활/호흡기 의료진(교육 콘텐츠 자문), SSC 심사위원(사회적 임팩트/포용성 관점)

## 4. Evidence and Confidence (Fact + Source + Confidence)

- Fact 1: CDC는 호흡기 바이러스 예방을 위해 최신 백신 접종, 실내 공기 개선, 손 위생 등을 권고함
- Source: CDC Respiratory Illnesses Prevention (2024-06-11)
- Confidence: High

- Fact 2: CDC는 혼잡한 실내, 환기 부족 공간, 본인/주변 고위험군이 있는 상황에서 마스크 착용을 권고함
- Source: CDC About Masks (2024-04-18)
- Confidence: High

- Fact 3: NHLBI는 COPD 예방/관리에서 금연, 간접흡연 회피, 오염원 회피를 제시함
- Source: NHLBI COPD Prevention (접근일: 2026-02-16)
- Confidence: High

- Fact 4: NHLBI는 COPD 환자에게 백신(독감, COVID-19, RSV, 폐렴구균 등)과 폐 재활, 운동 훈련을 제시함
- Source: NHLBI Living With COPD (접근일: 2026-02-16)
- Confidence: High

- Fact 5: WHO는 COPD 위험요인으로 흡연/간접흡연, 대기오염, 직업성 먼지/연기 등을 제시함
- Source: WHO COPD Fact Sheet (2024-06-16)
- Confidence: High

## 5. Assumptions

- 사용자는 긴 교육보다 1~2분짜리 짧은 예방 루틴을 더 꾸준히 수행할 것이다
- 보호자는 위기 시 텍스트보다 시각적 신호/버튼 기반 인터페이스에 더 빠르게 반응할 것이다
- 예방 행동의 "완료 체크"는 환자에게 통제감과 자기효능감을 제공할 것이다

## 6. Unknowns / Open Questions

- 한국의 실제 호흡기 환자/보호자에게 가장 실행이 어려운 예방 행동은 무엇인가?
- 위기 소통에서 가장 필요한 메시지 포맷(문장/아이콘/색상 코딩)은 무엇인가?
- 1주일 사용에서 이탈의 가장 큰 원인은 알림 피로인지, 조작 복잡도인지?

## 7. Root Cause Hypotheses and Validation Plan

- Hypothesis A: 예방 행동 실패의 핵심 원인은 지식 부족보다 "행동 전환 비용"이 높기 때문임
- Validation: 환자/보호자 5~8명에게 기존 행동 로그 인터뷰 + 하루 1분 미션 프로토타입 A/B 테스트(3일)

- Hypothesis B: 위기 상황 대응 지연의 핵심 원인은 "호흡 곤란 시 고비용 의사소통"임
- Validation: 1탭 도움 요청 UI vs 텍스트 입력 UI를 시나리오 테스트로 비교(과제 완료 시간 측정)

## 8. Impact Sizing (User + Business)

- User Impact: 예방 루틴 정착, 위기 불안 감소, 보호자의 대응 자신감 향상
- Project Impact(SSC): 사회적 영향(실사용 문제), 포용성(저호흡 상황 UI), 창의성(상황기반 인터랙션) 강화

## 9. Scope In / Out and Non-goals

- In Scope: 오프라인 동작 가능한 일일 예방 체크 루틴
- In Scope: 위기 시 1탭 소통 카드(예: "도움 필요", "지금 숨이 참")
- In Scope: 예방 행동 실패 시 대체 행동 제안(예: 짧은 실내 루틴)

- Out of Scope: 질병 진단/치료 결정을 자동화하는 기능
- Out of Scope: 실시간 병원 시스템 연동
- Out of Scope: 약물 용량/처방 변경 가이드

- Non-goal: 의료진을 대체하는 의학적 판단 도구를 만드는 것

## 10. Constraints and Dependencies

- SSC 심사 환경 특성상 네트워크 없이 핵심 흐름이 작동해야 함
- 의료 정보 표현은 공신력 가이드 기반으로 제한해야 함(과장/단정 금지)
- 짧은 개발 기간 내 구현 가능한 인터랙션 범위로 축소 필요

## 11. Success Metrics (North Star / Leading / Guardrail)

- North Star: `예방 루틴 5개 항목 중 일일 3개 이상 완료한 비율`
- North Star 목표: Baseline 40% -> Target 70% (2주 파일럿, 인앱 체크 이벤트 기준)

- Leading: `위기 소통 카드 과제 완료 시간` 20초 -> 8초 (사용성 테스트 기준)
- Leading: `일일 앱 재방문율` 1주 평균 45% 이상

- Guardrail: `불안 자기평가 점수(1~5)`가 사용 전 대비 1점 이상 악화되는 사용자 비율 10% 이하

## 12. Risks and Mitigations

- Risk 1: 의료 조언처럼 오해될 수 있음
- Mitigation: 의료 경고 문구 + "응급 시 지역 응급번호 사용" 안내 + 비진단성 표현 유지

- Risk 2: 알림 피로로 인한 이탈
- Mitigation: 알림 강도/시간 사용자 설정 + 알림 최소화 모드 제공

- Risk 3: 고령 사용자 조작 어려움
- Mitigation: 큰 터치 타깃, 단순 내비게이션, VoiceOver/Dynamic Type 우선 적용

1. `BreathBridge` (추천)

- 핵심 문제: 예방 행동의 낮은 지속성과 위기 소통 지연
- 핵심 기능: 60초 예방 루틴 카드(백신/손위생/마스크/환기/금연/활동)
- 핵심 기능: 위기 시 1탭 도움 요청 카드 + 보호자에게 보여줄 상태 화면
- 핵심 기능: 주간 리포트(오프라인)로 "실천한 예방 행동" 시각화

- 핵심 문제: 호흡 곤란 시 말하기 자체가 어려운 소통 단절
